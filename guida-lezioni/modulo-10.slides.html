Esercizio 1: Il gioco dell'impiccato

Il gioco dell'impiccato funziona così:

  • Il programma sceglie una parola segreta.
  • Il giocatore deve indovinare la parola lettera per lettera.
  • Ad ogni tentativo sbagliato si avvicina alla sconfitta.
  • Vince se scopre tutta la parola prima di esaurire i tentativi.

Passo 1 — Scegli una parola

Ci serve:

  • una lista di parole segrete (lette da un file?)
  • un modo per scegliere una parola casualmente
Dettagli
PAROLE = []

with open(file_input, "r", encoding="utf-8") as file_handler:
	for line in file_handler:
		PAROLE.append(line.strip())

parola = random.choice(PAROLE)
print(parola)

Prima di scrivere il codice, vediamo come si svolge il gioco passo per passo. La parola segreta è algoritmo (9 lettere).

INIZIO
  parola:   _ _ _ _ _ _ _ _ _
  errori:   0 / 6
  tentate:  —

Provo 'a'  →  INDOVINATA
  parola:   a _ _ _ _ _ _ _ _
  errori:   0 / 6
  tentate:  a

Provo 'z'  →  SBAGLIATA
  parola:   a _ _ _ _ _ _ _ _
  errori:   1 / 6
  tentate:  a  z

Provo 'o'  →  INDOVINATA
  parola:   a _ _ o _ _ _ _ o
  errori:   1 / 6
  tentate:  a  o  z

Provo 'l'  →  INDOVINATA
  parola:   a l _ o _ _ _ _ o
  errori:   1 / 6
  tentate:  a  l  o  z

  ... continua finché:
  - tutte le lettere sono scoperte  →  VITTORIA
  - gli errori raggiungono 6        →  SCONFITTA

Alcune osservazioni:

  • la lunghezza della parola è nota fin dall'inizio (quanti _?);
  • l'insieme delle lettere tentate cresce ad ogni turno;
  • l'errore si aggiunge solo se la lettera non è nella parola.

Passo 2 — Mostra la parola nascosta

Ci serve una funzione che riceve la parola e un insieme di lettere già indovinate, e restituisce una stringa con i caratteri scoperti e _ al posto di quelli ancora nascosti.

Esempio:

parola = "algoritmo"
lettere_indovinate = {"a", "o"}
-> "a _ _ o _ _ _ _ o"
Dettagli
def mostra_stato(parola, lettere_indovinate):

	nuova_stringa = ""
	for lettera in parola:
		if lettera in lettere_indovinate:
			nuova_stringa = nuova_stringa + lettera
		else:
			nuova_stringa = nuova_stringa + "_"

	return nuova_stringa

Passo 3 — Loop di gioco

Ad ogni passo del gioco, l'interfaccia:

  • mostra lo stato attuale;
  • mostra il numero di errori commessi fino a questo punto;
  • chiede una lettera all'utente;
  • controlla se è già stata tentata;
  • se no, controlla se la lettera è nella parola;
Dettagli
MAX_ERRORI = 6
lettere_indovinate = set()
lettere_tentate = set()
errori = 0

print(mostra_stato(parola, lettere_indovinate))
print(f"Errori: {errori}/{MAX_ERRORI}")

lettera = input("Indovina una lettera: ")

if lettera in lettere_tentate:
	print("Hai già tentato questa lettera.")

else:
	lettere_tentate.add(lettera)

	if lettera not in parola:
		errori += 1
		print(f"Sbagliato! La lettera '{lettera}' non c'è.")
	else:
		print(f"Brav*! La lettera '{lettera}' è presente.")
		lettere_indovinate.add(lettera)

Passo 4 — Condizioni di vittoria e sconfitta

Quando finisce il gioco?

  • se tutte le lettere della parola sono state indovinate -> vittoria;
  • se si esaurisce il numero di errori -> sconfitta.
Dettagli

while errori < MAX_ERRORI or not indovinato:
	...

Esercizio 2: Confronto di vocabolari

Hai due file di testo. Vuoi capire quanto sono simili in termini di vocabolario.

Scrivi un programma confronta.py che:

  1. Legge due file di testo passati come argomenti da linea di comando.
  2. Per ciascun file, costruisce:
    • il dizionario delle frequenze (parola -> conteggio);
    • il set del vocabolario (parole distinte).
  3. Stampa per ciascun file:
    • il numero totale di parole;
    • la dimensione del vocabolario;
    • le 5 parole più frequenti con il loro conteggio.
  4. Stampa le statistiche di confronto:
    • numero di parole in comune tra i due vocabolari;
    • numero di parole presenti solo nel primo file;
    • numero di parole presenti solo nel secondo file.

Esempio di output:

=== file1.txt ===
Parole totali: 1204
Vocabolario: 342 parole distinte
Top 5: the (98), of (72), and (65), to (54), a (48)

=== file2.txt ===
Parole totali: 876
Vocabolario: 289 parole distinte
Top 5: il (88), di (76), e (60), la (55), in (42)

=== Confronto ===
Parole in comune: 12
Solo nel primo file: 330
Solo nel secondo file: 277

Git e controllo di versione

Materiale adattato da: Version Control with Git — Software Carpentry, licenza CC BY 4.0.

Che cos'è il controllo di versione

Immagina di lavorare a un progetto nel tempo. Senza strumenti appositi, la cartella finisce per contenere:

relazione.txt
relazione_v2.txt
relazione_finale.txt
relazione_finale_def.txt
relazione_finale_def_vera.txt
relazione_finale_def_vera2.txt

Questo sistema è una soluzione manuale a un problema reale: vogliamo poter tornare indietro se sbagliamo qualcosa. Ma non funziona bene:

  • non si capisce quale sia la versione più recente;
  • non si sa cosa è cambiato tra una versione e l'altra;
  • se due persone lavorano sullo stesso file, i cambiamenti si sovrascrivono.

Il controllo di versione risolve questi problemi in modo sistematico. È un sistema che:

  • registra ogni modifica come una fotografia commentata (_snapshot_) dello stato del progetto;
  • conserva tutta la storia: si può tornare a qualsiasi versione precedente in qualsiasi momento;
  • tiene traccia di chi ha cambiato cosa e quando, rendendo la collaborazione possibile;
  • permette di lavorare in parallelo su parti diverse dello stesso progetto.
tempo ──────────────────────────────────────────────►

[c1]────[c2]────[c3]────[c4]────[c5]
  │        │       │       │       │
aggiunto  corretta aggiunto corrette HEAD
 input    media    output  stampe  (ora)

Ogni nodo è un commit: una fotografia dello stato del progetto in quel momento. Puoi sempre tornare a qualsiasi commit precedente.

Puoi pensarlo come un sistema di salvataggio automatico con memoria infinita e commenti. Non sostituisce solo tesi_finale_def_vera2.txt: sostituisce anche la cartella Dropbox/backup-20240312/, il tasto Ctrl+Z, e le email con allegati script_nuovo.py.

Git è il sistema di controllo di versione più diffuso al mondo. Lo usano singole persone per i propri progetti personali, team di ricerca, aziende con migliaia di sviluppatori, e praticamente ogni progetto software open source. Impararlo è una competenza trasversale: vale in informatica, in linguistica computazionale, in data science, in bioinformatica — ovunque si scriva codice o si gestiscano file di testo.

Git non è GitHub

Una distinzione importante prima di cominciare:

  • Git è il programma installato sul tuo computer che gestisce la storia del progetto;
  • GitHub è un sito web che ospita repository Git e facilita la collaborazione.

Puoi usare Git senza GitHub (lavorando solo in locale). GitHub diventa utile quando vuoi condividere il progetto, lavorarci da più computer, o collaborare con altre persone. GitHub è anche il posto dove vive la maggior parte del software open source del mondo.

Configurazione di Git

Prima di usare Git per la prima volta, bisogna configurare il proprio nome e indirizzo email. Queste informazioni vengono incluse in ogni commit.

git config --global user.name "Nomecognome"
git config --global user.email "indirizzo@email.com"

Per verificare la configurazione:

git config --list

L'opzione --global significa che la configurazione vale per tutti i progetti sul tuo computer. Puoi sovrascriverla per un singolo progetto omettendo --global.

Creare un repository

Un repository è una cartella il cui contenuto viene tracciato da Git.

Per trasformare una cartella esistente in un repository:

cd nome-cartella-progetto
git init

Esempio:

mkdir progetto-git
cd progetto-git
git init

Output tipico:

Initialized empty Git repository in /percorso/progetto-git/.git/

La cartella .git

Quando esegui git init, Git crea una cartella nascosta chiamata .git.

progetto-git/
└── .git/
    ├── config
    ├── HEAD
    ├── objects/
    └── refs/

Quella cartella contiene tutta la storia del progetto. Non va modificata a mano. Se la cancelli, perdi la storia.

Per verificare che Git stia monitorando la cartella:

git status

Output su un repository appena creato:

On branch main

No commits yet

nothing to commit (create/copy files and use "git add" to track)

Attenzione: non inizializzare un repository dentro un altro repository.
Se fai git init dentro una cartella già tracciata, Git si confonde.

Tracciare le modifiche

Il flusso di lavoro base di Git è sempre lo stesso:

modifica file -> git add -> git commit

Questi tre passi si ripetono continuamente. Ogni volta che arrivi a un punto stabile — hai aggiunto una funzione, corretto un bug, completato una sezione — fai un commit. La storia del progetto è la sequenza di questi commit.

Lo staging area: perché esiste?

Tra la modifica e il commit c'è un passo intermedio: lo staging area (area di preparazione).

Sembra una complicazione inutile, ma serve a qualcosa di preciso: puoi modificare dieci file ma committare solo quelli pronti, lasciando gli altri fuori dalla fotografia.

                   git add             git commit
cartella ─────────────────────► staging ──────────────► repository
di lavoro                        area                    (storia)
(file modificati)               (pronti)                 (commit)

                  git restore --staged
                ◄───────────────────────

Con git add scegli cosa includere nel prossimo commit. Con git commit salvi quella selezione come snapshot permanente. Con git restore --staged puoi togliere un file dallo staging area se hai cambiato idea.

Aggiungere file allo staging area

Crea un file e controllane lo stato:

echo "primo contenuto" > file.txt
git status

Output:

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        file.txt

Il file esiste ma Git non lo sta ancora tracciando. Per includerlo nel prossimo commit:

git add file.txt
git status

Output:

Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        new file: file.txt

Il file è nello staging area: è pronto per essere salvato.

Creare un commit

git commit -m "Aggiungo il primo file"

Output:

[main (root-commit) a1b2c3d] Aggiungo il primo file
 1 file changed, 1 insertion(+)
 create mode 100644 file.txt

Il commit è una fotografia commentata dello stato del progetto in quel momento.

Modificare un file

Modifica il file e controlla cosa è cambiato:

echo "secondo contenuto" >> file.txt
git status

Output:

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
        modified: file.txt

Per vedere esattamente cosa è cambiato:

git diff

L'output mostra le righe aggiunte (+) e quelle rimosse (-).

Per vedere le differenze di ciò che è già in staging:

git add file.txt
git diff --staged

Poi salva con un nuovo commit:

git commit -m "Aggiungo la seconda riga"

Messaggi di commit

Il messaggio di commit è la nota che accompagna ogni fotografia. Tra tre mesi, guardando la storia del progetto, è quello che ti dice perché hai fatto quella modifica — non solo cosa hai cambiato.

Un buon messaggio è breve, specifico, e scritto come un'azione:

  • Aggiungo il file di input
  • Correggo il calcolo della media
  • Aggiungo la gestione degli errori nel parsing

Evita messaggi generici come modifica, fix, aggiornamento, lavoro del 3 giugno. Quei messaggi non dicono nulla a chi legge la storia — incluso te nel futuro.

La regola pratica: il messaggio dovrebbe completare la frase _"Se applico questo commit, allora..."_

Esplorare la storia

Uno dei vantaggi principali di Git è che la storia del progetto è sempre consultabile. Puoi rispondere a domande come:

  • _"Quando ho introdotto questo bug?"_ → confronti versioni con git diff
  • _"Com'era questo file due settimane fa?"_ → recuperi con git restore
  • _"Cosa ho fatto ieri?"_ → leggi con git log

Per vedere la lista di tutti i commit:

git log

Output tipico:

commit b2c3d4e5f6... (HEAD -> main)
Author: Nomecognome <indirizzo@email.com>
Date:   Mon Jun 3 10:00:00 2026 +0200

    Aggiungo la seconda riga

commit a1b2c3d4e5...
Author: Nomecognome <indirizzo@email.com>
Date:   Mon Jun 3 09:00:00 2026 +0200

    Aggiungo il primo file

Versione compatta:

git log --oneline
b2c3d4e (HEAD -> main) Aggiungo la seconda riga
a1b2c3d Aggiungo il primo file

HEAD e la notazione ~

HEAD punta sempre al commit più recente. HEAD~1 è il commit precedente, HEAD~2 quello prima ancora.

a1b2c3d ────► b2c3d4e ────► c3d4e5f
    │               │             │
 HEAD~2          HEAD~1          HEAD
"Aggiungo      "Aggiungo      "Correggo
 il file"       la riga"       il bug"

Per confrontare la versione attuale con quella di due commit fa:

git diff HEAD~2

Per vedere il contenuto di un commit specifico:

git show HEAD~1

Recuperare una versione precedente

Per ripristinare un file a come era in un commit precedente:

git restore --source HEAD~1 file.txt

Oppure usando l'identificatore del commit:

git restore --source a1b2c3d file.txt

Il file nella cartella torna alla versione indicata. Per salvare questo cambiamento, fai un nuovo commit.

Ignorare file con .gitignore

Alcuni file non devono essere tracciati da Git:

  • file temporanei e compilati (.pyc, __pycache__/);
  • file di configurazione locale con dati sensibili;
  • file di output generati automaticamente.

Crea un file .gitignore nella radice del repository:

*.pyc
__pycache__/
.DS_Store
output/

Ogni riga è un pattern. Git ignorerà i file che corrispondono.

Per verificare quali file vengono ignorati:

git status --ignored

Una volta creato .gitignore, aggiungilo al repository:

git add .gitignore
git commit -m "Aggiungo .gitignore"

Repository remoti e GitHub

Finora abbiamo lavorato in locale: la storia del progetto esiste solo sul nostro computer. Questo va bene per progetti personali, ma ha limiti evidenti:

  • se il computer si rompe, si perde tutto;
  • non si può lavorare comodamente da più dispositivi;
  • non si può collaborare con altre persone.

La soluzione è un repository remoto: una copia del repository ospitata altrove, su un server raggiungibile da internet.

   computer locale                   GitHub (remoto)
┌───────────────────────┐       ┌──────────────────────┐
│  progetto/            │       │  utente/progetto      │
│  ├── script.py        │       │  ├── script.py        │
│  └── .git/            │       │  └── .git/            │
│      └── storia       │       │      └── storia       │
└───────────────────────┘       └──────────────────────┘
            │                               │
            └──────── git push ────────────►│
            │◄──────── git pull ────────────┘

GitHub è la piattaforma più usata per ospitare repository Git. Oltre a fare da backup remoto, GitHub offre:

  • un'interfaccia web per navigare il codice e la storia dei commit;
  • strumenti per la collaborazione (pull request, code review, issue tracking);
  • la possibilità di rendere il progetto pubblico e contribuire a progetti open source;
  • un portfolio: mostrare i propri repository GitHub è ormai normale nei CV tecnici.

Esistono alternative (GitLab, Bitbucket, Codeberg), ma GitHub è il più diffuso.

  • Git è il programma di controllo versione che usiamo sul computer;
  • GitHub è il servizio online che ospita i repository.

Puoi usare Git anche senza GitHub.

Creare un repository su GitHub

  1. Accedi a GitHub.
  2. Clicca su New repository.
  3. Dai un nome al repository.
  4. Lascia vuoto (senza README) se vuoi collegarlo a un repository locale esistente.
  5. Clicca su Create repository.

Collegare il repository locale al remoto

Dopo aver creato il repository su GitHub, copia l'URL e collega il remoto:

git remote add origin https://github.com/utente/nome-repository.git
git push -u origin main
  • git remote add origin ... registra il repository remoto con il nome origin;
  • git push pubblica i commit locali sul remoto;
  • -u origin main imposta il ramo remoto di default per i push futuri.

Per verificare i remoti configurati:

git remote -v

Sincronizzare con il remoto

Dopo aver aggiunto altri commit locali:

git push

Per scaricare aggiornamenti dal remoto (utile se hai lavorato da un altro computer):

git pull

Collaborare

Fino a qui abbiamo sempre lavorato da soli su un repository nostro. Il vero potere di Git emerge quando più persone lavorano sullo stesso progetto.

Il meccanismo è semplice: ognuno ha una copia locale del repository, lavora sulla propria copia, e periodicamente sincronizza con il remoto. Git si occupa di unire i cambiamenti di tutti.

Aggiungere un collaboratore su GitHub

Il proprietario del repository deve aggiungere l'altra persona come collaboratore:

  1. Vai alla pagina del repository su GitHub.
  2. Clicca su Settings → Collaborators.
  3. Cerca il nome utente GitHub del collaboratore e invitalo.
  4. Il collaboratore riceve un'email e deve accettare l'invito.

Dopo l'accettazione, entrambe le persone possono fare push sul repository.

Clonare un repository

Chi vuole lavorare su un repository esistente lo clona in locale:

git clone https://github.com/utente/nome-repository.git

Questo crea una copia locale completa del repository, inclusa tutta la storia. Non serve git init: il repository è già inizializzato.

Il flusso collaborativo

Ogni persona lavora così:

pull → modifica → add → commit → push

In dettaglio:

git pull               # scarica i cambiamenti del collega
# ... modifica i file ...
git add lista.txt
git commit -m "Aggiungo le verdure"
git push               # pubblica i tuoi cambiamenti

La regola fondamentale è: prima pull, poi push.

Perché? Perché se il tuo collega ha già pubblicato dei commit, il tuo repository locale è indietro rispetto al remoto. Git rifiuterà il push finché non ti aggiorni. Fare pull prima evita questo problema — e nella maggior parte dei casi Git riesce a unire automaticamente i cambiamenti senza conflitti, purché non abbiate modificato le stesse righe degli stessi file.

               GitHub (remoto)
                    │
         push ▲     │ ▼ pull
                    │
         Persona A  │  Persona B
         (locale)       (locale)

Esercizio: la lista della spesa in coppia

Formate coppie. Una persona crea il repository, l'altra collabora.

Persona A — crea il repository:

  1. Crea un nuovo repository su GitHub chiamato lista-spesa (pubblico, senza README).
  2. Sul proprio computer:
mkdir lista-spesa
cd lista-spesa
git init
git branch -M main
  1. Crea il file lista.txt con due o tre voci:
pane
latte
uova
  1. Aggiungi, committa e collega il remoto:
git add lista.txt
git commit -m "Prima versione della lista"
git remote add origin https://github.com/personaA/lista-spesa.git
git push -u origin main
  1. Aggiungi Persona B come collaboratore (Settings → Collaborators).

Persona B — clona e modifica:

  1. Accetta l'invito via email.
  2. Clona il repository:
git clone https://github.com/personaA/lista-spesa.git
cd lista-spesa
  1. Apri lista.txt e aggiungi altre voci (frutta, verdura, ecc.).
  2. Committa e fai push:
git add lista.txt
git commit -m "Aggiungo frutta e verdura"
git push

Persona A — aggiorna e aggiungi:

  1. Scarica i cambiamenti di Persona B:
git pull
cat lista.txt     # verifica che le voci di B siano presenti
  1. Aggiungi ancora qualcosa, committa e fai push.

Tutti e due:

Leggete la storia del repository con git log --oneline. Vedrete i commit di entrambi nella stessa sequenza.

Parte 2 — Provocare e risolvere un conflitto:

Ora vediamo cosa succede quando due persone modificano la stessa riga.

Senza fare pull, entrambe le persone aprono lista.txt e modificano la prima riga della lista con due valori diversi.

Persona A scrive:

pane integrale

Persona B scrive:

pane di segale

Persona A committa e fa push per prima:

git add lista.txt
git commit -m "Cambio tipo di pane"
git push

Persona B committa e prova a fare push:

git add lista.txt
git commit -m "Scelgo pane di segale"
git push

Git rifiuterà il push con un messaggio del tipo:

 ! [rejected]  main -> main (fetch first)
hint: Updates were rejected because the remote contains work that you do not have locally.

Persona B deve fare pull:

git pull

A questo punto Git segnala il conflitto nel file lista.txt:

<<<<<<< HEAD
pane di segale
=======
pane integrale
>>>>>>> a1b2c3d...

Persona B risolve il conflitto:

  1. Apri lista.txt nell'editor.
  2. Decidi quale riga tenere (o scrivi una versione condivisa, es. pane).
  3. Rimuovi i marcatori <<<<<<<, =======, >>>>>>>.
  4. Salva il file.
  5. Committa la risoluzione:
git add lista.txt
git commit -m "Risolvo conflitto sul tipo di pane"
git push

Persona A fa pull per ricevere la risoluzione:

git pull

Tutti e due:

Eseguite git log --oneline e osservate la storia completa: vedrete il commit di A, il commit di B, e il commit di risoluzione del conflitto.

Conflitti

Lavorando in collaborazione, prima o poi succede: due persone modificano la stessa parte dello stesso file in modo incompatibile.

Git non può decidere da solo quale versione è quella "giusta" — dipende dall'intenzione delle persone che hanno fatto le modifiche. Quindi segnala il conflitto e lascia la decisione a te.

I conflitti sembrano spaventosi la prima volta, ma sono un meccanismo di sicurezza: Git si rifiuta di sovrascrivere silenziosamente il lavoro di qualcuno. Preferirebbe bloccarsi e chiederti cosa fare, piuttosto che perdere delle modifiche.

storia condivisa       A modifica      B modifica
                        riga 3          riga 3
[c1]──[c2]──[c3]        │               │
              │          ▼               ▼
              ├────── [cA] "latte"   [cB] "succo"
              │
           entrambi fanno push:
           il secondo trova un CONFLITTO
           perché c3 → cA e c3 → cB sono incompatibili

Come si presenta un conflitto

Quando git pull trova modifiche incompatibili, il file coinvolto contiene:

<<<<<<< HEAD
versione tua (locale)
=======
versione dell'altro (remota)
>>>>>>> a1b2c3d4...

Come risolverlo

  1. Apri il file in un editor.
  2. Decidi quale versione tenere (o scrivi una versione che unisce entrambe).
  3. Rimuovi i marcatori <<<<<<<, =======, >>>>>>>.
  4. Salva il file.
  5. Esegui:
git add file.txt
git commit -m "Risolvo il conflitto"

Il conflitto è risolto e la storia del progetto è integra.
